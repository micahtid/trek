1. **Add integration dependencies:**
   ```bash
   flutter pub add flutter_secure_storage oauth2_client
   ```
   `flutter_secure_storage` is for storing the GitHub OAuth token securely (Keychain on iOS, Keystore on Android).
   `oauth2_client` provides `GitHubOAuth2Client` for the GitHub OAuth flow.

2. **Replace `lib/features/settings/settings_screen.dart`** with the full Settings screen:

   Layout (top to bottom):
   - **Profile section** at the top:
     - User avatar (CircleAvatar with the user's Google profile picture, or initials fallback)
     - Display name in Sora bold
     - Email address in lighter weight
     - Read profile from `ref.watch(authNotifierProvider)` — the `AuthStateAuthenticated` has `displayName`, `email`, `avatarUrl`

   - **Integrations section** (labeled "Connections" or "Integrations"):
     - Google Calendar connection — delegate to `CalendarAuthSection` widget
     - GitHub connection — delegate to `GitHubAuthSection` widget

   - **Account section** at the bottom:
     - Sign Out button (destructive style — red text or outlined in red)
     - On tap: `ref.read(authNotifierProvider.notifier).signOut()`

   Use a `ListView` with `ListTile`-style items for the integration rows. Keep the design clean, light, and consistent with the amber/Sora theme.

3. **Create `lib/features/settings/calendar_auth_section.dart`**:

   A widget that:
   - Shows the current Calendar connection status (connected/not connected)
   - Shows a "Connect Google Calendar" button when not connected
   - Shows "Connected" with a checkmark when the scope has been granted
   - On tap: Calls `GoogleSignIn.instance.authorizeScopes(['https://www.googleapis.com/auth/calendar.readonly'])`
   - On success: Updates a local state or provider to reflect the connection
   - On failure: Shows a SnackBar with the error

   **CRITICAL per research:**
   - Do NOT use `signIn()` with scopes — use `authorizeScopes()` for incremental scope (Pitfall 5)
   - This is separate from the primary sign-in flow — it adds a scope to the existing session
   - Calendar scope is `https://www.googleapis.com/auth/calendar.readonly` — read-only, not write
   - The scope grant status can be persisted locally (SharedPreferences or Riverpod state) — it does not need to go to Convex. The actual Calendar API calls happen in Phase 3.

4. **Create `lib/features/settings/github_auth_section.dart`**:

   A widget that:
   - Shows the current GitHub connection status (connected/not connected)
   - Shows a "Connect GitHub" button when not connected
   - Shows the GitHub username and "Connected" when connected
   - On tap: Triggers GitHub OAuth flow via `oauth2_client`
   - On success: Stores the token in `flutter_secure_storage`, updates UI
   - On failure: Shows a SnackBar with the error

   GitHub OAuth setup:
   ```dart
   final ghClient = GitHubOAuth2Client(
     redirectUri: 'com.interngrowthvault://oauth2redirect',
     customUriScheme: 'com.interngrowthvault',
   );
   final helper = OAuth2Helper(
     ghClient,
     grantType: OAuth2Helper.authorizationCode,
     clientId: GITHUB_CLIENT_ID, // Constant — user will set this
     clientSecret: GITHUB_CLIENT_SECRET, // Constant — user will set this
     scopes: ['read:user'], // Only read:user for Phase 1 — repo scope deferred to Phase 5
   );
   ```

   **Per research Open Question 4:** Request only `read:user` scope in Phase 1. The `repo` scope (for commits/PRs) is added in Phase 5 when actually needed — following the same incremental authorization pattern as Calendar.

   **CRITICAL per research (Pitfall 6):** GitHub OAuth uses `oauth2_client` which is completely independent of `google_sign_in`. They operate through different mechanisms. Verify during testing that completing GitHub OAuth does NOT disrupt the Google sign-in session.

   The GitHub Client ID and Client Secret should be defined as constants in a config file or at the top of the section. The executor should note in the SUMMARY that the user must create a GitHub OAuth app and set these values.

   Store the access token in `flutter_secure_storage` with a key like `github_access_token`. On app start (or when Settings is opened), check if a token exists to show the connected state.

   Optionally, after connecting, make a test API call to `https://api.github.com/user` to get and display the GitHub username.

**Per user decisions:**
- Both Calendar and GitHub connections live in Settings only — NOT in onboarding
- Both are optional — the app works without either
- Clean, light design consistent with the rest of the app

**Do NOT:**
- Prompt for Calendar or GitHub during sign-in or onboarding
- Request `repo` scope for GitHub — that is Phase 5
- Store the Google ID token or any Google secrets — google_sign_in manages its own tokens
- Use SharedPreferences for GitHub token — use flutter_secure_storage (per research Don't Hand-Roll table)

**User setup required (documented for executor):**
- GitHub OAuth App: User must create a GitHub OAuth App at https://github.com/settings/developers
  - Set callback URL to `com.interngrowthvault://oauth2redirect`
  - Copy Client ID and Client Secret into the app's config constants
- Google Cloud Console: Calendar API must be enabled in the Google Cloud project for Calendar scope to work
